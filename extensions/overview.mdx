Extensions are self-contained packages that add new features to the web UI. Each extension is a directory placed inside `extensions/` (for user-installed extensions) or `extensions-builtin/` (for those shipped with the web UI). When the web UI starts, it scans both directories, loads each extension's Python scripts, registers their UI components, and wires up any callbacks they declare.

Extensions differ from standalone scripts in scope and integration depth. A script lives in a single `.py` file under `scripts/` and attaches to the generation pipeline via the `Script` base class. An extension can include multiple scripts, JavaScript files, a stylesheet, a `metadata.ini` manifest, and an `install.py` dependency installer — giving it the ability to add entirely new tabs, inject settings sections, hook into the sampling loop, and ship its own static assets.

## Extension directory structure

A typical extension looks like this:

```
extensions/
└── my-extension/
    ├── scripts/
    │   └── my_script.py      # One or more Script subclasses
    ├── javascript/
    │   └── my_script.js      # Optional JavaScript loaded into every page
    ├── style.css             # Optional stylesheet injected into the UI
    ├── install.py            # Optional dependency installer run at startup
    └── metadata.ini          # Extension manifest (name, requirements, etc.)
```

The `scripts/` directory is required for extensions that participate in generation. The web UI scans it for subclasses of `modules.scripts.Script` and `modules.scripts.ScriptPostprocessing` and registers them automatically. All other directories and files are optional.

## How extensions differ from scripts

| | Scripts | Extensions |
|---|---|---|
| Location | `scripts/*.py` | `extensions/<name>/scripts/*.py` |
| JavaScript | Not supported | `javascript/*.js` loaded automatically |
| Stylesheets | Not supported | `style.css` loaded automatically |
| New UI tabs | Not supported | Via `on_ui_tabs` callback |
| Dependency installer | Not supported | `install.py` run at startup |
| Settings entries | Not supported | Via `on_ui_settings` callback |
| Enabled/disabled via UI | Not supported | Yes, from the **Extensions** tab |

## Popular community extensions

The community has built hundreds of extensions on top of these APIs. Some of the most widely used include:

<CardGroup cols={3}>
  <Card title="ControlNet" icon="sitemap">
    Adds conditioning inputs (depth maps, edge maps, pose skeletons, etc.) that give precise structural control over generations.
  </Card>
  <Card title="AnimateDiff" icon="film">
    Converts the image generation pipeline into a video generator by inserting a temporal motion module into the UNet.
  </Card>
  <Card title="ADetailer" icon="user">
    After Detailer — automatically detects faces and other regions using object detection models, then re-inpaints them at higher detail.
  </Card>

These extensions are installed from their respective GitHub repositories via the **Extensions** tab. See [Installing extensions](/extensions/installing) for instructions.
